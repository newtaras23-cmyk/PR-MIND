# SEO + AEO Audit — PR-MIND

**Дата:** 2026-07-04
**Обсяг:** локалі `en`/`uk` (34 сторінки × 2 = усі поточні маршрути). `fr`/`es` — поза скоупом (свідоме рішення з попереднього заходу, сайт наразі має лише 2 локалі).
**Метод:** production-збірка (`next build && next start`), craul власним Node-скриптом (curl+regex-парсинг `<title>`/meta/canonical/hreflang/H1/JSON-LD по всіх 34 сторінках), структурна валідація JSON-LD проти вимог schema.org/Rich Results, Lighthouse CLI v11.7.1 (v12 несумісний із Node 20 в цьому середовищі — аудит `canonical` падає з `URL.parse is not a function`).
**Немає інтернет-доступу до живого Google Rich Results Test / validator.schema.org** (домен `prmind.example` — плейсхолдер, сайт не задеплоєний) — структуровані дані валідовано за офіційними вимогами полів schema.org програмно, а не через live-інструмент. Це явно позначено нижче.

**Що виправлено (з доказом до/після):**

1. **SEO-категорія показувала помилку на кожному прогоні** (`URL.parse is not a function` в аудиті `canonical`, специфічно для lighthouse@12 + Node 20). Це проблема інструменту, не сайту. Використано `lighthouse@11.7.1` (сумісний з Node 20) — SEO-категорія тепер рахується: **100/100** на всіх 6 прогонах. Додатково: canonical при тестуванні на `localhost` не збігався з захардкодженим `https://prmind.example` в метаданих — це коректна поведінка для продакшену, але заважала аудиту на localhost. `siteUrl` зроблено env-конфігурованим (`NEXT_PUBLIC_SITE_URL`) — на реальному домені canonical природньо збігатиметься з serving origin.

2. **Accessibility 85-86 на сторінках з формами** (`LeadForm`, `CallbackForm`):
   - `label`/`select-name`: `<label>` не був прив'язаний до `<input>`/`<select>` через `htmlFor`/`id` → додано пари `htmlFor`/`id` на всіх полях форм; для inputs без видимого label (CallbackForm) додано `aria-label`
   - `color-contrast`: білий текст на фоні `--action` (#0A84FF) = **3.65:1** (потрібно 4.5:1 WCAG AA) — системний дефект на **всіх** заповнених CTA-кнопках сайту (7 місць у 5 файлах). Замінено фон на `--action-dim` (#0A5DB0) = **6.55:1** (пройдено з запасом), hover — `brightness-90` замість зміни кольору
   - `label-content-name-mismatch`: `aria-label` перемикача мов повністю ховав видимий текст "EN"/"UK" → тепер aria-label включає видимий текст
   - Результат: Accessibility **86 → 100** на пілларі/статті, **95/96 → 100** на головній

3. **Performance mobile-home 78 → 86, LCP 6.1s → 3.9s** (−36%), total page weight 558KB → 417KB (−25%), unused JS 140KB → 27KB (−81%):
   - GSAP+ScrollTrigger (`ReputationSurface`) і Lenis (`SmoothScrollProvider`) статично імпортувались у загальний бандл на кожній сторінці → переведено на `dynamic import()` всередині ефектів (контент лишається SSR для SEO, бібліотеки вантажаться асинхронно після гідратації)
   - Framer Motion повний API (`motion.div`) → `LazyMotion`+`domAnimation`+`m.div` (стандартний патерн зменшення бандла)
   - `<Link href="/contacts">` на кожній сторінці (Header, Footer, Hero, CTA-смуга) автоматично префетчив важкий бандл `/contacts` (форми + zod, ~70KB) навіть на головній, де форми не використовуються → `prefetch={false}` на цих посиланнях

4. **Головна на mobile (Performance 86, LCP 3.9s) — залишається нижче цілі ≥95/<2.0s.** Корінь підтверджено `mainthread-work-breakdown`: ~1.8s реальної роботи головного потоку під 4× CPU throttling через гідратацію ~18 окремих `Reveal`-анімацій + GSAP-секції "Reputation Surface" — усе це прямо передбачено ТЗ §3.5/§3.6 (stagger при завантаженні, scroll-reveal, signature-секція). П'ять з шести інших комбінацій сторінка/пристрій — 94-100, LCP<2s. **За рішенням користувача — прийнято як відомий, задокументований ліміт**, а не приховано; подальше покращення вимагало б скорочення кількості анімацій, що суперечило б явним вимогам ТЗ.

### Рекомендації на майбутнє

1. **Реальні шрифти Clash Display / General Sans** — зараз тимчасова заміна (Bricolage Grotesque / Inter), бо ліцензійних файлів немає в npm-реєстрі й немає прямого доступу для завантаження з fontshare.com з цього середовища. Замінити, щойно файли будуть надані.
2. **Реальне лого + профілі соцмереж** — щойно з'являться, додати `Organization.logo`/`sameAs` — це підвищить якість Knowledge Panel і впізнаваність сутності для AI.
3. **Зображення-обкладинки для статей Insights** — підвищить eligibility для Rich Results image та привабливість у соцмережах (OG-image).
4. **Теми статей з найбільшим потенціалом AI-цитування** (виходячи з того, що AI-пошук найохочіше цитує конкретні, структуровані, "як зробити X" відповіді): "Скільки коштує прибрати негативний відгук з Google" (конкретна, часто шукана AEO-тема), "Критерії значущості (notability) для сторінки у Вікіпедії для СЕО/засновника", "Чим відрізняється видалення від приглушення (suppression) негативного контенту" (роз'яснення термінів — улюблений формат AI Overviews), порівняльна стаття "ORM vs SEO vs PR — у чому різниця" (порівняльний формат часто цитується Perplexity/ChatGPT).
5. Після деплою на реальний домен — перевірити Lighthouse/PageSpeed Insights на живому URL: mobile-lab throttling у пісочниці цього середовища є навмисно жорстким синтетичним профілем (4× CPU + повільна мережа), і фактичний UX на реальних 4G/5G/WiFi пристроях користувачів, ймовірно, суттєво кращий за лабораторні цифри.
